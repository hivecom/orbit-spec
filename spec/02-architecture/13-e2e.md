# End-to-End Encryption

The E2E protocol design. Where E2E applies and where it doesn't - the
transport-vs-E2E boundary - has one home in
[Messaging](10-messaging.md#security-model-transport-vs-end-to-end); this
page specifies the protocol for the contexts where it does.

## Scope

E2E encryption in Orbit applies to exactly two contexts:

- **1:1 DMs** (messages between two registered users)
- **P2P calls** (direct WebRTC connections negotiated over IRC tags)

It intentionally doesn't apply to channels or group voice:

- **Channels** are server-mediated by design - history, search, and threads
  all require the server to read content. A channel is a room the server
  runs; if you join it, you trust the operator. Encrypting a channel would
  silently break every server-side feature users depend on.
- **Group voice** routes media through an SFU, which must be able to forward
  the streams. If you need a private call with no server in the path, use
  P2P - that's exactly what it's for.

E2E-encrypted DMs coexist with the server's standard DM storage: the server
stores and delivers ciphertext with the same retention and delivery
mechanics as plaintext, offline delivery and history work, and server-side
search doesn't. See [Messaging - Storage](10-messaging.md#storage).

## Protocol

Double Ratchet (the Signal protocol) via the maintained Rust implementation
of libsignal:

- Well-proven in production at scale.
- Forward secrecy: past messages are safe even if the current key is
  compromised.
- Post-compromise security: the ratchet heals after a key compromise.

MLS (RFC 9420) is out of scope. It's designed for group E2E, and since group
E2E isn't in scope, it adds complexity with no benefit.

## Key Transport

The Double Ratchet needs an initial key exchange, and Orbit uses the existing
IRC connection as the key exchange channel:

1. Both users publish their identity public keys and one-time prekeys via the
   metadata extension. Keys are server-visible, which is fine - public keys
   are public.
2. On the first DM, the initiating client performs an X3DH key agreement
   using the recipient's published identity key and a one-time prekey.
3. The initial encrypted message carries the ratchet header as a tag on the
   message; subsequent messages carry the ratchet state the same way.
4. The recipient's client completes the X3DH agreement, derives the same root
   key, and the ratchet is established.

Encrypted messages are transported as normal IRC messages whose body is
ciphertext, tagged with the sender's key ID so the recipient can select the
correct ratchet state. A conversation's E2E status is client-derived from
those tags, with a visible marker when encryption is first established
([Messaging - Storage](10-messaging.md#storage)). The tag definitions live in
[Implementation - Tags](../03-implementation/02-tags.md).

## P2P Calls

P2P WebRTC connections are E2E-secure by construction, with no additional
application-layer encryption:

- DTLS-SRTP is mandatory for all WebRTC media; the browser and Tauri stacks
  enforce it.
- The IRC handshake carries the DTLS fingerprints, which both peers verify at
  connection time - a man-in-the-middle can't substitute their own
  certificate without failing the fingerprint check.
- Post-handshake, the server isn't in the media path at all
  ([Satellite - 1:1 Calls](05-satellite.md#11-calls-p2p)).

## Cross-Device Key Synchronization

A user with multiple devices needs their E2E keys synchronized. Three
approaches, in order of complexity - the choice between them is genuinely
undecided and tracked in [Open Questions](../OPEN-QUESTIONS.md):

**Option A: key backup to Depot.** Conversation keys are encrypted with a
recovery passphrase and stored as an object in Depot. A new device
authenticates via OIDC, fetches the encrypted bundle, prompts for the
passphrase, and decrypts. Works offline and uses existing infrastructure
(Depot + OIDC); the passphrase is a UX burden, and losing it means losing
access to E2E history.

**Option B: device-to-device transfer via P2P.** Adding a new device opens a
P2P connection from an existing device (the same handshake mechanism as
calls) and transfers keys over the encrypted data channel, verified by a
short code shown on both screens. No passphrase, but both devices must be
online simultaneously and the first device is a single point of failure.

**Option C: per-device keys (the Signal model).** Each device has its own
key pair, the user publishes a device list via metadata, and senders encrypt
to all of a recipient's known devices. Devices are independent and there's
no passphrase or transfer step; message size scales with device count,
device management (revoking, trusting) is real UX complexity, and it's the
most complex to implement.

## Risks

- **Key verification.** How does a user confirm they're talking to the right
  person and not an impersonator with a different key? Safety number
  comparison and QR scanning are the candidates; this needs UX prototyping,
  because it only helps if users actually do it.
- **Key recovery after device loss.** Without a backup or another device, the
  ratchet state is gone and prior messages can't be decrypted - E2E means the
  server can't help. Key backup is the primary mitigation.
- **Anonymous users.** E2E requires stable identity (persistent key pairs).
  Guests have none and can't participate in E2E DMs; the client must make
  clear that E2E DMs require a registered account.
- **Transitions.** A conversation that starts unencrypted can transition to
  E2E once both clients negotiate keys. The reverse is technically possible
  but should be discouraged by the client, with a clear warning if encryption
  is being disabled.

## Dependencies

- The metadata extension must be available - key publication relies on it.
- Stable account registration and SASL authentication
  ([Identity](09-identity.md)).
